1、为什么要对象转json? 序列化。 java对象在内存中，如果想传输到硬盘或者其他地方，以对象的方式可能不太方便，这时就需要转换为json形式

6、redis实现分布式session 第四期 ： 1小时50分 session共享： 2小时05
因为session只能存在一台服务器上，当我们有多个服务器时，用户请求到了别的服务器进行登陆时将造成获取不到用户信息session的情况，所以这时候就需要一个共享存储。 将session存入redis后，无论用户请求哪个服务器都不影响session的识别了
为什么要使用redis来存?
答： 因为redis基于内存，， 比起存在硬盘里的数据库来说读写性能高出很多倍

List collect = list.stream().map(user -> userService.getSafetyUser(user)).collect(Collectors.toList());

9、定时任务插入数据 第六期40分 ， 普通循环插入，也就是将插入写在循环里很慢，用mp分批次插入时20秒10万条
10、线程池 第六期 1小时40分
问： 什么情况下会超过默认线程数？答：任务队列满了之后就会增加线程数，让更多的人来干活。就会默默的增加到最大线程数的数量
11、推荐逻辑实现 第六期1小时50
为何需要缓存？ 数据量大，查询的时候太慢了，会给用户一个不好的体验，可以先把那些需要频繁查询的数据查出来，放到缓存里，这样就优化很多了也就不需要查数据库了。
②如果说先在数据库里查完再放入缓存里的话，那么第一个用户使用体验就很差了，这时我们可以使用预加载缓存，如何预加载？ 可以定时更新这个缓存。在一个用户少的时间定时更新（定时任务）
③这时就要用到分布式锁，为何要用到分布式锁，应用场景是什么？ 假如我们同一份代码是部署在多个机器上，假如说0点到了，该更新缓存了，那难道我们每个机器都要去执行一次这个任务吗？这个时候就可以用到分布式锁，控制同一时间只有一台机器去执行定时任务，其他机器就不用重复执行了。
④分布式锁的应用场景：比如说抢票，总共只有五张票，假如说同时有五个用户分别请求了不同的机器，然后这些机器先去查这个票数，发现查询时确实是有五张票，然后这五个用户每人都买五张票，那最后肯定是有人买不到的，这时就需要用分布式锁控制一下，这时就只能是一个服务器访问一个共享资源，而不是说所有的用户集中查这个库存，以保证每次读票查票的时候都是一个最新的状态。
分布式锁是一种用于控制分布式系统中多个节点之间对共享资源的访问的技术。它可以确保在任何给定时刻只有一个节点能够访问共享资源，从而避免竞态条件和数据不一致性问题。
12、查询使用分页查询时：数据库有十万条数据时用分页查询每页展示20条数据查询时间是90毫秒，当我的数据是一百万条时查询时间就变为了1.19秒？ 再使用Redis缓存查询时变成了22.54毫秒 提高了20倍
13、缓存 7期20分。 Redis
如果缓存放在两台服务器上，可能会造成数据不一致，这时就需要使用分布式缓存，实现方式：redis分布式缓存。
问：redis是什么？
答：NoSql数据库，K-V键值对存储系统
问:redis的实现方式？
答：Spring Data Redis :通用的数据访问框架，定义了一组增删改查的接口
如果项目是分布式的，需要用到一些分布式的特性，分布式锁，分布式集合，推荐用Redisson来操作Redis
问:Redis 数据结构有哪些？
答: ① String字符串类型 ： name :”PN”
②List 列表类型 : names:[“s”,”wang”,”s”] 数组有长度，列表无,列表可有重复值
③ Set集合： names:[“zhang”,”wang”] 值不能重复
④ Hash 哈希 ： nameAge:{“tao”:18,”zhang”:20} 相当于里面存了个map，里面的键不能重复”tao”
⑤Zset集合：names:{tao - 9 , zhang - 12} 9分，12分 必须有一个分数
opsForValue：用于操作 Redis 中的字符串类型（String）数据。
opsForHash：用于操作 Redis 中的哈希类型（Hash）数据。
opsForList：用于操作 Redis 中的列表类型（List）数据。
opsForSet：用于操作 Redis 中的集合类型（Set）数据。
opsForZSet：用于操作 Redis 中的有序集合类型（ZSet）数据。

14、给推荐模块加缓存 7期 1小时18分
①Redis缓存得设置过期时间，因为要是一直存在的话，明明数据库都更新了，因为是优先查缓存的关系，到后面还是一直查旧数据。
②Ridis内存不能无限增加，一定要设置过期时间。如果不设置过期时间，缓存中的数据将永久存在，可能会导致内存耗尽，影响系统的稳定性和性能。
15、缓存预热 7期1小时33分
问：为什么要用缓存预热？
答：①因为如果是查完数据库之后再将数据写入缓存中，那么就会面临一个问题，那就是总有那么一个第一次打开页面开始查的，这时还没有缓存，它的体验就会很差。
②如果是双十一这种在短时间会有大量用户涌进来的情况，在一瞬间大量的用户同时使用这个数据库，那么可能会造成崩溃。 

缓存预热可能存在的问题：
预热优点：对于开发者：使用预热能一定程度上保护数据库。对于用户：让用户始终访问很快
    预热缺点：①增加开发成本  ② 预热的时机和时间如果错了，有可能缓存的数据不对可能是太老了
                举个例子啊，本来这个用户访问的时候才缓存数据，是不是就是能够拿到当时用户第一次访问的那个数据库的最新的状态，但是如果你提前预热，比如说我提前七天预热，是不是用户最后访问到的数据是七天前的数据对不对？也要存在这样一个问题啊，所以不是什么时候都要预热的。
          ③占用空间 ：起初有4个G，本来上线后就占用了2个G，再预热就又占用了2个G ，那这就是拿空间换时间，不值得。

怎么缓存预热？
①定时 ② 模拟触发（手动触发）
定时任务实现方式：Spring Scheduler （spring boot 默认整合）
用定时任务，每天刷新所有用户的推荐列表。

答：（新增用户少，但是总用户量多）
②缓存的空间不能太大，要留给其他缓存空间
③ 缓存数据的周期（一天一次）

17、分布式锁 第8期 15分钟
引出： 多台服务器同时执行定时任务向数据库插入数据时可能会出现大问题，比如数据重复插入，数据竞争，这时就需要用上分布式锁。 所以要控制定时任务在同一时间只有一个服务器能执行。
使用分布式锁来确保在多台服务器之间同一时间只有一个服务器能够执行插入操作。这可以通过工具如Redis的分布式锁来实现。
好几台服务器，但只有一个锁，谁抢到这个锁谁就能执行定时任务。
锁：有限资源的情况下，控制同一时间段只有某些线程（用户/服务器）能访问到资源。
Java 实现锁: synchronized 关键字、并发包的类
缺点:只对单个JVM有效，也就是说每个商场都有自己的厕所，自己抢自己的了。

为啥需要分布式锁?
1.有限资源的情况下，控制同一时间 (段) 只有某些线程(用户/服务器)访问到资源
2.解决单个锁只对单个JVM有效的情况

抢锁机制： 8期 50分
核心思想就是:先来的人先把数据改成自己的标识(服务器 ip)，后来的人发现标识已存在，就抢锁失败，继续等待。
等先来的人执行方法结束，把标识清空，其他的人继续抢锁。

Redisson实现分布式锁。8期一小时50分

Redisson在加锁和释放锁的时候都自动实现了lua脚本 ，实现了原子操作。

OpenAPI

API签名认证

2期1小时.20分

通过请求头传递(secretKey不能放在请求头中)

为什么需要这个东西，因为当我们做好了一个接口时，在没有做任何认证的情况下任何人都可以调用，这样可能会导致安全问题，有的人可能会恶意调用。

所以需要一个accessKey(标识)和secretKey(密钥)

类似于用户名和密码但是是无状态的，也就是每一次都得带着这个东西过来进行验证。

密钥不能够在服务器之间进行传递，有可能会被拦截

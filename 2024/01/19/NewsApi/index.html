MD5签名可解密

签名/摘要算法，主要是为了防止数据被更改

加密算法和签名算法有什么区别?

  • 简单的说，签名算法是摘要的这种类型就是说它是保证数据不被篡改，但是并没法根据计算后的这个值还原回原来的这个值，所以要加盐，而加密的话是要还原回来的。

传输时传输要加密的字符，加密时要在服务端加盐值来混合加密，即便被拦截到了要加密的字符他发送过来还是不会跟服务端的匹配上

别人是怎么解密的？

  • 他们将所有能够想到的字符串进行一个加密然后将加密过的值保存到数据库中，然后通过你发过去的加密值和他们数据库里的加密值进行对比，如果能够匹配上，就能反向推出加密前的值。

在网络上传输好像不太安全了，但是对于本项目来说还可以，性能也不错

例如啊，我给服务端传了一个爱学啊。然后我把爱学啊这几个字母通过md 5或者说类似于md 5的这种签名算法计算出来一个值。然后我把这两个值都给它传到服务端去，那如果在传输的过程中，有人把爱学啊这几个字母更改了一位，或者说增加了一位，或者删除了一位。那么我服务端拿到这个签名和爱学啊更改后的这几个字母计算一下，如果计算出来的这个签名和客户端传递过的这个签名不一样，那就表示这个文件。或者说表示这个用户登录信息在传输过程中被更改了，对吧？也就是说签名的这种算法一般就是这样一个用途。但是一定要注意，但是如果别人知道我用的是md 5，它更改了以后再签名一下，也就是再用md 5签名一下，那我就没法追踪到是不是我？客户端传的数据还是被人家攻击了，传的数据所以说在用这种签名摘要算法的时候，还要给它加一点盐，所谓的加一点盐，

你可以理解为就是说在正常的数据里面。我们混入一点，我们写死的数据，这样的话，如果别人没有把你的应用破解了，他就不知道你混入的数据是什么，虽然他知道你是md 5，但是他拿着艾雪儿edu这几个字母去签名。他就不知道。虽然他知道是md 5，但是他拿着I学re du这几个字母再去md五一下，它的值就和我们计算的不一样好

SHA256更加安全

非对称加密

RSA

每次加密数据都不一样，使用公钥加密，私钥解密。

非对称加密有公钥和私钥两个概念，私钥自己拥有，不能给别人，公钥公开。根据应用的不同，我们可以选择使用不同的密钥加密：

  1. 签名：使用私钥加密，公钥解密。用于让所有公钥所有者验证私钥所有者的身份并且用来防止私钥所有者发布的内容被篡改，但是不用来保证内容不被他人获得。
  2. 加密：用公钥加密，私钥解密。用于向公钥所有者发布信息,这个信息可能被他人篡改,但是无法被他人获得。

Sa-Token

对于一些登录之后才能访问的接口（例如：查询我的账号资料），我们通常的做法是增加一层接口校验：

  • 如果校验通过，则：正常返回数据。
  • 如果校验未通过，则：抛出异常，告知其需要先进行登录。

那么，判断会话是否登录的依据是什么？我们先来简单分析一下登录访问流程：

  1. 用户提交 name + password 参数，调用登录接口。
  2. 登录成功，返回这个用户的 Token 会话凭证。
  3. 用户后续的每次请求，都携带上这个 Token。
  4. 服务器根据 Token 判断此会话是否登录成功。

所谓登录认证，指的就是服务器校验账号密码，为用户颁发 Token 会话凭证的过程，这个 Token 也是我们后续判断会话是否登录的关键所在。

Sa-Token 为这个账号创建了一个Token凭证，且通过 Cookie 上下文返回给了前端。

Redis

使用Sa-Token集成的Redis实现分布式Token登录

Sa-Token 默认将数据保存在内存中，此模式读写速度最快，且避免了序列化与反序列化带来的性能消耗，但是此模式也有一些缺点，比如：

  1. 重启后数据会丢失。
  2. 无法在分布式环境中共享数据。

为此，Sa-Token 提供了扩展接口，你可以轻松将会话数据存储在一些专业的缓存中间件上（比如 Redis）， 做到重启数据不丢失，而且保证分布式环境下多节点的会话一致性。

Spring事务的应用

在用户新增收货地址时，如果它本身已经有了一个默认收货地址，此时想覆盖掉起初的收货地址时，那么正确的操作应该是将之前的默认收货地址取消，然后再添加最新的收货地址设置为默认。但是问题是，此时需要操作两次数据库，第一是先将原本的默认收货地址设置为0，第二次再试更新收获地址。但是这时就出现了事务问题了，可能会将默认收货地址取消之后出现异常，此时就是没有默认收获地址了，也没有添加新的收货地址成功。

Spring事务需要出现异常才能够回滚，什么情况下事务会失效。

  • 事务是关系型数据库才有的
  • 它的原理是对需要处理的方法进行一个cry catch处理，try里面如果没有异常，那么就不会回滚，出现了异常就在catch里面回滚事务。

1、异常捕获处理但不抛出

  • 理论上出现异常那么事务就会回滚，但是如果把这个异常try catch捕获但不抛出就可能导致事务失效。
  • 原因：事务通知只有捉到了目标抛出的异常，才能进行后续的回滚处理，如果目标自己处理掉异常，事务通知无法知悉。

1.RSA签名算法

使用RSA非对称加密算法时,起初为了方便，创建了一个RSA工具类。只需要传一个要加密的数据便可以加密数据，然后使得一个RSA类变量为静态变量，然后再工具类的构造器中为其赋值，考虑到这个工具类生成的RSA对象不是单例对象，使用了单例设计模式，然后在外部直接通过类变量的方式去使用RSA加密密码,但是试了很多次都发生了失败，直接通过全局异常处理器返回给前端。当时没想那么多，试了很多次都这样，通过创建单例对象的方式去调用RSA的时候idea提示我通过类.变量名要好一点我就听取了建议，我就以为是可以这样的。DEBUG模式也没有报空指针异常，因为我配置了全局异常处理器的原因，有的信息直接返回给前端了，但是没返回是什么异常，到最后才发现是因为我这个变量是静态变量并且没有为其进行赋值，所以在通过类.变量名调用的时候不会创建RSA工具类中的构造方法去创建这个对象为rsa变量赋值，导致为空无法进行加密操作。

不是单例的话会导致公钥私钥不一样，到时候无法用私钥解密

至于如何证明new 一个RSA对象它生成的两个不同的公钥私钥，我是在加密的时候NEW了一次，在解密的时候NEW了一次，发现根本无法解密，然后用单例的方式就解决了

2.RAS工具类

在 Kotlin 中，lazy 函数是一种惰性初始化的委托属性。它接收一个 lambda 表达式，这个 lambda 表达式会在首次访问属性时被调用，返回的值会被存储，并在后续的访问中直接返回。

在你提供的代码中，val rsa: RSA by lazy { RSA() } 声明了一个只读的属性 rsa，它的值是通过 lazy 委托初始化的。这意味着 RSA() 的实例只在首次访问 rsa 属性时创建，而后续的访问都直接返回这个创建好的实例，实现了懒加载的效果。
